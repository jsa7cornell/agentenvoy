# Multi-user context for dashboard redesign

**Date:** 2026-04-24
**From:** Track 1 (partial-attendance) backend work

The dashboard's "thread card" is no longer implicitly 1:1. A link can now have N named invitees (`NegotiationLink.inviteeNames: string[]`), and the session can be configured with `partialAttendance: "allowed"` + a `minimumAttendees` threshold — meaning a meeting can confirm even if some invitees are busy, with the excused ones getting a fallback email (recording/summary). Today the ThreadCard signals multi-invitee with a small 👥 count chip in the title row and a title like "Surf (Will, Andrew, Bo)" instead of "Surf: Will + John". Participant status (agreed / active / pending / declined) already renders as colored dots in a sub-row on group events. The host needs to see *at a glance* which invitees have engaged, who's declined, and whether quorum is met — this is distinct from "is the meeting confirmed" and neither state is well-represented today. When you redesign, assume the card sometimes represents a roster, not a person.

Two things worth keeping in mind: (1) the guest picker will soon surface **per-slot invitee availability** (see `previews/multi-invitee-slot-picker.html` for three concepts — roster row / quorum chip / per-slot ruler dots) so the host dashboard should probably mirror whatever visual vocabulary we land on; consistency across host+guest sides matters. (2) The underlying data is already plural end-to-end — `SessionInvitee` (stable invite) + `InviteeSlotRsvp` (per-slot RSVP) + `SessionParticipant` (actual arrivals) are three different things. Most 1:1 affordances today (single avatar, "Waiting for Sarah" pill, "invitee in CEST" chip) either need a plural variant or should gracefully degrade when invitee count > 1. Scenarios worth pressure-testing are in `previews/partial-attendance-scenarios.md`.
