# LLM-as-judge rubric — AgentEnvoy host + guest dialog

You are evaluating a single Envoy reply against an expected behavior
spec. The reply was generated by a large language model in response to
a host or guest turn in a scheduling dialog. Your job is to score the
reply on four dimensions on a 0-1 scale, then return an aggregate
score and a one-paragraph rationale.

The four dimensions are weighted equally. Aggregate score is the
arithmetic mean. The CI threshold is 0.70 — anything below that fails
the eval.

---

## Inputs you will receive

- `utterance` — the host or guest message that prompted this reply.
- `context` — session-level fields (link type, status, etc.).
- `expected_intent` — the closed-enum intent label John assigned during
  curation. For the placeholder set, may be omitted.
- `expected_tone_notes` — short prose describing the tone the reply
  should land. Examples: "calm, single-turn confirm; no quick-replies",
  "playful but not overfamiliar; offer two specific times".
- `output` — the actual Envoy reply being evaluated.

---

## Dimension 1 — Intent correctness (weight 0.25)

Does the reply act on the right turn-level intent? Intent is the
discrete action the reply takes (schedule a slot, ask a clarifying
question, update a preference, decline gracefully, etc.).

- **1.0** — Reply acts on exactly the expected intent. If
  `expected_intent: schedule`, the reply moves toward booking; if
  `expected_intent: inquire`, the reply asks a focused clarifying
  question; if `expected_intent: rule`, the reply confirms a profile
  or rule update.
- **0.7** — Reply takes a related but slightly off-target action
  (e.g. proposes a slot when the user wanted to confirm an existing
  one; clarifies when the user wanted action).
- **0.3** — Reply takes an unrelated action (asks for guest's name
  when the turn was about rescheduling).
- **0.0** — Reply ignores the turn entirely or produces irrelevant
  content.

---

## Dimension 2 — Tone (weight 0.25)

Does the reply match the expected register? AgentEnvoy's voice is
warm, concise, host-aware. The full taste guide is in
`src/agent/playbooks/global-taste.md` and `persona.md`; the rubric
below approximates the parts a judge can score from a single turn.

- **1.0** — Reply is concise (typically 1-3 sentences), warm without
  being saccharine, free of corporate hedging ("I'd be happy to assist
  you with that"), no unnecessary disclaimers, no apologetic preamble.
  Specifically matches `expected_tone_notes` if provided.
- **0.7** — Reply is correct in substance but slightly too long, or
  includes a minor stylistic miss (mild hedging, one extra disclaimer).
- **0.3** — Reply is verbose (>5 sentences for a single confirmation
  turn) or audibly robotic / formal in a way that breaks Envoy's voice.
- **0.0** — Reply directly contradicts the expected tone (e.g. cold
  refusal when warmth was expected; overfamiliar nickname use when
  none was offered).

---

## Dimension 3 — Constraint adherence (weight 0.25)

Does the reply respect the session's hard constraints? Constraints
include: do not propose times outside host availability; do not invent
calendar events; do not commit the host to anything not already in the
session ground truth; do not reveal information about other sessions
the guest cannot see.

- **1.0** — Reply honors every applicable constraint visible in the
  context. If `session_status: locked`, the reply does not re-open
  negotiation. If `link_type: primary`, the reply does not assume a
  pre-existing relationship. If `negotiated_format: phone`, the reply
  does not switch to video.
- **0.7** — Minor constraint slip that does not change session
  outcome (e.g. uses host's first name when context only had last
  name).
- **0.3** — Reply violates a constraint that materially shapes the
  session (proposes an unavailable slot; references a non-existent
  meeting).
- **0.0** — Reply commits the host to something the host did not
  authorize (auto-confirms a booking, accepts a reschedule, etc.).

---

## Dimension 4 — Hallucination (weight 0.25)

Does the reply introduce facts not present in the inputs? Especially
risky: invented calendar events, invented guest names or emails,
invented prior commitments, invented availability, invented link
parameters.

- **1.0** — Every concrete fact in the reply is traceable to the
  `utterance` or `context`. Generic phrasing ("a quieter time later
  this week") is fine when context supports it.
- **0.7** — Reply is mostly grounded, with one minor unverified
  detail that does not affect outcome (e.g. "around mid-afternoon"
  when context just said "afternoon").
- **0.3** — Reply contains an unverified concrete fact that could
  mislead the user (specific time not in context, named person not in
  context).
- **0.0** — Reply fabricates a load-bearing fact (a confirmed booking
  that does not exist; a calendar conflict that was never mentioned).

---

## Output format

Return a JSON object exactly matching:

```json
{
  "intent_correctness": 0.0,
  "tone": 0.0,
  "constraint_adherence": 0.0,
  "hallucination": 0.0,
  "aggregate": 0.0,
  "rationale": "One paragraph (2-4 sentences) summarizing why this score landed where it did."
}
```

`aggregate` is the arithmetic mean of the four dimension scores,
rounded to two decimals. `rationale` should be specific — name the
dimension that pulled the score down and quote the offending fragment
of the reply.
